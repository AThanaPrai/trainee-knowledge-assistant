# แผนการ Onboard Trainee 5 คน ตลอด 4 เดือน

---

## สัปดาห์แรก: วางรากฐาน

**วันที่ 1-2: ปฐมนิเทศและทำความรู้จัก**
- พา trainee ทุกคนทำความรู้จักกับทีมและระบบที่ใช้งานจริง (codebase, tools, workflow)
- อธิบาย culture ของทีม: วิธีการสื่อสาร, การถามคำถาม, การรับ feedback
- **Set expectation ตั้งแต่วันแรก** — บอกให้ชัดว่าใน 4 เดือนนี้จะถูกประเมินอะไร (technical skills, communication, ownership) และเป้าหมายคืออะไร ให้รู้ว่าตัวเองกำลังเดินไปทางไหน

**วันที่ 3-5: งานจริงชิ้นแรก**
- ให้ศึกษาการใช้งาน tool ที่จะใช้
- มอบ task เล็กๆที่มีขอบเขตชัดเจนให้แต่ละคนทำคนเดียว ที่เกี่ยวข้องกับ tool ที่ให้ศึกษา
- เป้าหมายคือให้รู้สึกว่าตัวเองทำได้ เพิ่มความมั่นใจ
- Review งานร่วมกันเน้นอธิบายว่าทำแบบนี้ทำไม

---

## เดือนที่ 1: เน้นความคุ้นเคย

- มอบ task ที่ยากขึ้นเรื่อยๆ ทุกสัปดาห์ โดยยังมี scope ชัดเจน
- จับคู่ (pair) trainee แต่ละคนกับงาน feature จริงในโปรเจกต์
- ให้ trainee review งานกันเองและรับ feedback จากทีม
- เปิดโอกาสให้ถามโดยไม่รู้สึกอาย สร้างความคุ้นเคยใน team
- ส่งเสริมให้ถามเมื่อไม่รู้ — พี่เองก็ไม่ได้รู้ทุกอย่าง แต่รู้ว่าจะไปหาคำตอบได้จากที่ไหน อยากให้ trainee มีทัศนคติแบบเดียวกัน: ไม่รู้ไม่เป็นไร แต่ต้องรู้จะถามใครหรือหาจากไหน
    - ถ้า trainee ไม่กล้าถาม — พี่ก็ถามก่อนเลย ไม่ได้ถามแค่เรื่องงาน แต่คุยเรื่องทั่วไปด้วย เพื่อให้รู้สึกสบายใจก่อน พอไว้วางใจแล้ว update งานก็จะไหลออกมาเองโดยไม่รู้สึกว่ากำลังถูก monitor
    - ถ้า trainee ซ่อนปัญหา กลัวดูไม่เก่ง — อธิบายให้เห็นว่าการแจ้งปัญหาเร็วคือสิ่งที่ทีมต้องการงานที่ค้างอยู่คนเดียวครึ่งวัน อาจคลี่คลายได้ในไม่กี่นาทีถ้าพูดออกมา
    - ถ้า trainee ไม่แจ้งล่วงหน้าว่างานจะไม่ทัน — ชี้ให้เห็นว่าปัญหาไม่ได้อยู่ที่งานไม่เสร็จ แต่อยู่ที่ทีมไม่มีโอกาสปรับ plan ล่วงหน้า การแจ้งก่อนคือการรับผิดชอบต่องานที่ทำ

- ตั้ง 1-on-1 รายสัปดาห์กับแต่ละว่าคนงานถึงไหน เรียนรู้อะไรบ้าง
- หลังคุยเสร็จให้ trainee สรุปสิ่งที่คุยกัน — สิ่งที่ได้รับมอบหมาย, สิ่งที่ต้องทำ, สิ่งที่ยังติดอยู่ — แล้วส่งกลับมาให้พี่ดู เพื่อให้แน่ใจว่าเข้าใจตรงกัน และมีหลักฐานว่าสั่งอะไรไปแล้ว

**วัดผลเดือน 1:**
- ส่ง task ตรงเวลาได้ 80% ขึ้นไป
- present งานที่ทำให้คนอื่นฟังได้
- ไม่ต้องถามคำถามเดิมซ้ำสองครั้ง

---

## เดือนที่ 2: เพิ่มความรับผิดชอบ

- ให้แต่ละคนเป็นเจ้าของ feature เล็กๆ ตั้งแต่ design ถึง deploy
- เมื่อ feature เสร็จให้ present ให้ทีมฟัง — อธิบายว่าทำอะไร ตัดสินใจอะไร และเจอปัญหาอะไรระหว่างทาง
- เริ่มทำงานกับผู้ที่เกี่ยวข้องจากนอกทีม เช่น ประสานงานกับ PM, QA หรือลองคุยกับลูกค้าโดยตรงในบางกรณี เพื่อฝึก communication กับคนที่ไม่ได้มีพื้นหลัง technical เหมือนกัน
- 1-on-1 เปลี่ยนโฟกัสจาก "งานเสร็จไหม" เป็น "เรียนรู้อะไรเพิ่ม"

**วัดผลเดือน 2:**
- สามารถ own feature จนจบได้โดยไม่ต้องให้ตามตลอด
- มีส่วนร่วมใน team discussion อย่างน้อย 1 ครั้งต่อสัปดาห์
- คุณภาพงานดีขึ้นวัดจาก review comments ที่ลดลง
 
---

## เดือนที่ 3: ทำงานอิสระ

- ลองให้ trainee วางแผนงานของตัวเองและคาดการณ์เวลาที่จะทำได้
- เปิดโอกาสให้เสนอ idea หรือ improvement ในระบบที่ตัวเองดูแล

**วัดผลเดือน 3:**
- วางแผนงานได้แม่นภายใน ±20%
- มี initiative เสนอปรับปรุงสิ่งที่เห็นว่าควรแก้ไข
- สามารถทำงานข้ามวันโดยไม่ต้องถาม blocker ทุกชั่วโมง

---

## เดือนที่ 4: ประเมินและตัดสินใจ

- สรุปผลงานตลอด 4 เดือน ร่วมกับแต่ละคน
- ประเมิน 3 ด้าน: technical skills, communication, ownership
- ให้ trainee ทำ self-assessment ก่อน เพื่อดูว่ารับรู้ตัวเองตรงกับที่ทีมเห็นไหม

**เป้าหมายสุดท้ายที่วัดได้:**
- **ผลงาน:** deploy อย่างน้อย 3 feature หรือ task สำคัญต่อคน ตลอด 4 เดือน — นับเฉพาะงานที่ขึ้น production จริง ไม่นับงานที่ค้างอยู่ใน branch
- **คุณภาพ:** code review pass rate ≥ 70% — PR ที่ส่งอย่างน้อย 7 ใน 10 ครั้ง merge ได้โดยไม่ต้องแก้ใหม่ครั้งใหญ่
- **พัฒนาการ:** สามารถทำงานที่ยากกว่าเดือนแรก 2 เท่าได้โดยไม่ต้องช่วยตลอดเวลา — วัดจากขนาดและความซับซ้อนของ task ที่รับได้เทียบกับเดือนแรก
- **Retain:** อย่างน้อย 3 ใน 5 คนอยากอยู่ต่อและทีมอยากเก็บไว้ — วัดทั้งสองทาง ไม่ใช่แค่ trainee อยากอยู่ แต่ทีมต้องอยากได้ด้วย

---

## จัดการกับ Low Performers

ถ้า trainee คนใดไม่ผ่านเกณฑ์เดือนที่ 1-2:

1. **คุยตรงๆ ก่อน** — ทำความเข้าใจ และถามว่ามีปัญหาอะไรที่ทำให้ติดขัด บางครั้งปัญหาคือสภาพแวดล้อม ไม่ใช่ความสามารถ อาจจะยกตัวอย่างประสบการณ์ตัวเองที่ยังเป็น trainee
2. **แยก task ใหญ่เป็น task เล็ก** — ถ้าติดเพราะ task ใหญ่เกินไปจนไม่รู้จะเริ่มตรงไหน ให้ช่วยแบ่งออกเป็นขั้นตอนย่อยที่ทำได้ใน 1-2 ชั่วโมง ช่วยให้เห็น progress ทีละก้าว ไม่ใช่จมอยู่กับงานที่ดูใหญ่เกินไป
3. **ตั้ง Performance Plan 2 สัปดาห์** — กำหนดเป้าหมายชัดเจน เช่น "ส่ง task นี้ให้เสร็จภายในวันศุกร์"
4. **ถ้ายังไม่ดีขึ้น** — แจ้งหัวหน้าทีมและพิจารณาเปลี่ยน scope งาน หรือสรุปว่า fit ไม่ตรง แจ้งตรงๆ ดีกว่าปล่อยให้ทั้งสองฝ่ายเสียเวลา

เป้าหมายคือ retain 3/5 — ไม่ใช่ retain ทุกคนไม่ว่าจะเป็นอย่างไร การปล่อยคนที่ไม่ fit ออกไปเร็วก็เป็นการตัดสินใจที่ดีสำหรับทั้งสองฝ่าย

---

## สรุป

ผมว่าผมเป็นคนที่ให้ความสำคัญเรื่อง communication เป็นอันดับหนึ่งครับถ้าทำงานกันในทีมแต่ละคนก็ควรรู้ว่าแต่ละคนมีหน้าที่อะไร ทำอะไรกันอยู่ ติดปัญหาตรงไหนไหม ถ้ามีจะได้รีบช่วยกันแก้ ไม่ควรจะกลัวว่าคนอื่นจะมองเราว่าไม่เก่ง เป็นตัวถ่วง รีบแก้ไข รีบทำงานให้เสร็จอย่างมีคุณภาพเวลาพักจะได้พักอย่างสบายใจ
