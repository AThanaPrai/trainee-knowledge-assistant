# Architecture Decisions

## Decision 1: แยก Server/Client Component

แยกแต่ละหน้าออกเป็นสองไฟล์ — `page.tsx` รันบน server ทำหน้าที่ดึงข้อมูลจากฐานข้อมูลและตรวจสอบว่า user login อยู่ไหม แล้วส่งข้อมูลที่ได้ไปให้ `ChatClient.tsx` ที่รันบน browser เพื่อจัดการทุกอย่างที่ user โต้ตอบได้

เหตุผลคือถ้าให้ browser ดึงข้อมูลเอง user จะเห็นหน้าว่างก่อนแล้วค่อยโหลด ซึ่งช้ากว่า และต้องเปิด API เพิ่มโดยไม่จำเป็น ยิ่ง API มากยิ่งมีโอกาสเขียนผิดและเป็นช่องโหว่ได้ การแยกแบบนี้ทำให้ข้อมูลพร้อมตั้งแต่หน้าโหลด และข้อมูลสำคัญอย่างรหัสผ่านหรือ database connection ไม่มีโอกาสหลุดออกไปถึง browser เลย

---

## Decision 2: ใช้ Hash Embedding แทน Embedding API สำหรับ RAG

แทนที่จะเรียก embedding API ภายนอก (OpenAI, Cohere ฯลฯ) เลือกทำ embedding function ใน local โดยใช้ FNV-1a hashing บน word frequency เพื่อสร้าง vector 128 มิติแล้วเก็บใน Chroma

เหตุผลตรงๆ คือตอน build ไม่มี API key สำหรับ embedding โดยเฉพาะ เลยต้องหาทางทำให้ระบบรันได้ก่อนโดยไม่ต้องพึ่ง service ภายนอก วิธีที่เลือกคือนับความถี่ของคำในแต่ละ chunk แล้วแปลงเป็น vector ที่ใช้เปรียบเทียบความใกล้เคียงระหว่างเอกสารกับคำถามได้

วิธีนี้จับคู่ได้ทั้ง keyword ที่ตรงกันและความใกล้เคียงของเนื้อหาในระดับหนึ่ง เพราะเอกสารที่พูดถึงเรื่องเดียวกันมักใช้คำกลุ่มเดียวกัน vector ที่ได้จึงใกล้กัน แต่ยังด้อยกว่า model จริงที่เข้าใจความหมายของประโยค เช่น ถามว่า "ราคาเท่าไหร่" อาจไม่ดึง chunk ที่พูดถึง "ค่าใช้จ่าย" ออกมาได้ เพราะคำไม่ตรงกัน

ถ้ามี API key อยากเปลี่ยนไปใช้ OpenAI text-embedding ซึ่งเข้าใจความหมายได้ดีกว่ามาก โดยแก้แค่ฟังก์ชัน `embed()` ใน `chroma.ts` ไฟล์เดียว ส่วนอื่นทั้ง upload และ chat ไม่ต้องแตะเลย

---

## Decision 3: รองรับ AI หลายเจ้าในที่เดียว

แทนที่จะผูกกับ AI provider เจ้าเดียวตลอด สร้าง function `chat()` ที่เป็นตัวกลาง — ส่วนอื่นของระบบเรียก `chat()` แล้วได้คำตอบกลับมา โดยไม่ต้องรู้ว่าข้างหลังใช้ Claude, OpenAI, Gemini, Groq หรือแม้แต่ mock สำหรับทดสอบ การเปลี่ยน provider ทำได้แค่แก้ environment variable

เหตุผลคือระหว่าง develop ใช้ mock ทำให้รันได้โดยไม่ต้องมี API key และถ้า provider หนึ่ง quota หมดหรือแพงเกินก็เปลี่ยนได้ทันทีโดยไม่ต้องแก้โค้ด ข้อเสียคือมีชั้นกลางเพิ่มขึ้นหนึ่งชั้น แต่เพราะทุก provider รับและส่งข้อมูลในรูปแบบคล้ายกัน ความซับซ้อนที่เพิ่มมาจึงน้อยมาก
